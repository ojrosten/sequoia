      11              :     \brief Implementation details for semantics checks that cleanly supports types which do/do not have allocators.
      12              : 
      13              :     The general pattern in this file is of paired function templates of the form
      14              : 
      15              :     <pre>
      16              :     template<test_mode Mode, class Actions, movable_comparable T, class... Args>
      17              :     ret_type do_check_foo(std::string_view, test_logger<Mode>& logger, const Actions&, const T&, const T&, const Args&...);
      18              : 
      19              :     template<test_mode Mode, class Actions, movable_comparable T>
      20              :     ret_type check_foo(std::string_view description, test_logger<Mode>& logger, const sentinel<Mode>& sentry, const Actions& actions, const T& x , const T&y)
      21              :     {
      22              :       return do_check_foo(description, logger, actions, x, y);
      23              :     }
      24              :     </pre>
      25              : 
      26              :     The idea is that the implementation of check_foo in this file is appropriate for types without allocators.
      27              :     As such, check_foo delegates to the instantiation of do_check_foo for which sizeof...(Args) is zero. However,
      28              :     do_check_foo is designed such that it can also deal with types which may allocate. As such, do_check_foo takes
      29              :     a template argument 'Actions'. Appropriate realizations of this type possess a bunch of constexpr static bools
      30              :     which indicate whether or not additional actions should be carried out.
      31              : 
      32              :     For example, when checking the semantics of a regular type, two const references to the type must be supplied.
      33              :     It is a prerequisite that these instances are not equal to one another. This is always checked. However, if
      34              :     the type allocates, clients may prefer to utilize the allocation-checking extension supplied with sequoia. In
      35              :     this case, there is a constexpr flag which indicates that, after checking the prerequisite, it is further checked
      36              :     that <pre>operator==</pre> hasn't unwittingly allocated - as can happen if it accidentally captures by value rather than reference.
      37              : 
      38              :     Thus, the general structure is
      39              : 
      40              :     <pre>
      41              :     some_common_check(...);
      42              :     if constexpr (Actions::some_flag)
      43              :     {
      44              :       do_some_extra_check(...);
      45              :     }
      46              :     </pre>
      47              : 
      48              :     where the extra check is fed, amongst other things, the parameter pack args...
      49              : 
      50              :     Therefore when checking the semantics of an allocating type, the skeleton of the function is the same as for
      51              :     non-allocating types. However, in the former case extra checks are folded in. This pattern ensures
      52              :     consistency: if the scheme is tweaked for non-allocating types, allocating types will automatically benefit.
      53              :     But there is enough flexbility to allow for all the extra allocation checks that allocating types require.
      54              : 
      55              :     As such, the various check_foo functions in this header have overloads in AllocationCheckersDetails.hpp which
      56              :     supply the extra argument(s) for the additional actions that comprise allocation checking.
      57              : 
      58              :     Note that check_foo / do_check_foo pairs in this file are common to both regular and move-only types.
      59              :     However, regular types additionally have copy semantics; the extra pieces necessary for this may be found
