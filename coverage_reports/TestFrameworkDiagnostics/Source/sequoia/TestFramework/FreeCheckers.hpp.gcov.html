      10              : /*! \file
      11              :     \brief Free functions for performing checks, together with the 'checker' class template which wraps them.
      12              : 
      13              :     Given a type, `T`, any reasonable testing framework must provide a mechanism for checking whether or
      14              :     not two instances of `T` are, in some sense, the same. If the type implements `operator==` then it
      15              :     is natural to utilize this. However, there is much more to the story. First of all, if this check
      16              :     fails then, in order to be useful, there must be some way of serializing the state of `T`. This may
      17              :     be done by specializing the class template \ref serializer_primary "serializer" for cases where
      18              :     `operator<<` is not appropriately overloaded.
      19              : 
      20              :     However, there is an alternative which may be superior. Consider trying to implement `vector`.
      21              :     This class has various const accessors suggesting that, if `operator==` returns `false`, then the
      22              :     accessors can be used to probe the exact nature of the inequality. To this end, a class template
      23              :     \ref value_tester_primary "value_tester" is defined. Specializations that provide an implementation
      24              :     for the static method `void test(equality_check_t, ...)` perform a detailed check of the equality
      25              :     of the state of two supposedly equal instance of a class. Note that there is no need for the
      26              :     value_tester to include checks of `operator==` or `operator!=`, since these will be done,
      27              :     regardless. Indeed, the value_tester will be triggered independently of whether `operator==`
      28              :     fails since either the latter or the accessors may have bugs, and introducing unnecessary dependencies
      29              :     would reduce the fidelity of the testing framework.
      30              : 
      31              :     If a value_tester is supplied, clients may typically expect a notification that `operator==` has
      32              :     returned `false`. There will then be a series of subsequent checks which will reveal the precise nature
      33              :     of the failure. For example, for vectors, one will be told whether the sizes differ and, if not, the
      34              :     element which is causing the difference between the two supposedly equal instances. If the vector holds
      35              :     a user-defined type then, so long as this has its own value_tester, the process will continue until a type
      36              :     is reached without a value_tester; typically this will be a sufficiently simple type that serialization is
      37              :     the appropriate solution (as is the case for may built-in types).
      38              : 
      39              :     Suppose a client wishes to compare instances of `some_container<T>`. If `some_container<T>` has a
      40              :     specialization of \ref value_tester_primary "value_tester" then this will be used; if it does
      41              :     not then reflection is used to determine if `some_container` overloads `begin` and `end`.
      42              :     If so, then it is treated as a range and all that is required is to implement a `value_tester` for `T`
      43              :     (unless serialization is prefered). If `some_container` is user-defined, it is wisest to provide an overload
      44              :     of the \ref value_tester_primary "value_tester". However, if the container is part of std, it is probably
      45              :     safe to assume it works correctly and so instead effort can be directed focused on T.
      46              : 
      47              :     With this in mind, imagine creating a container, `C`. One of the first things one may wish to do is to check
      48              :     that it is correctly initialized. It would be a mistake to use `value_tester<C>::test(equality_check_t, ...)`
      49              :     for this since, to do so, a second, identical instance would need to be created. This is circular and prone to
      50              :     false positives. However, one may instead furnish `value_tester<C>` with methods
      51              :     `test(equivalence_check_t, ...)` and/or `test(weak_equivalence_check_t, ...)`.
      52              :     
      53              :     We may consider a value for e.g. `std::vector<int>` to be equivalent to an  `initializer_list<int>` in
      54              :     the sense that they hold (at the relevant point of the program) the same elements. Thus, an appropriate
      55              :     defintion of `test(equivalence_check_t, ...)` is supplied. Of course, there is more to a vector than the
      56              :     values it holds: there is the entire allocator framework too. In this case, however, it is not part of
      57              :     the logical structure and, indeed, the state of the allocator is not considered in `vector<T>::operator==`.
      58              :     Thus it is philosophically reasonable to consider equivalence of `vector` and `initializer_list`. However,
      59              :     sometimes is is useful to check the equivalence of the state of an instance of `T` to a proper subset of
      60              :     the logical state of an instance of some S. For this purpose, an appropriate defintion of
